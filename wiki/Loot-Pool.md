# Loot Pool

**Sources:** `modules/routes/api/lootpool.py`, `modules/routes/api/base_pool_blueprint.py`, `modules/services/base_pool_service.py`, `modules/repositories/lootpool_repo.py`, `modules/repositories/base_pool_repo.py`

## Overview

The loot pool system tracks weekly item drops available in Wynncraft's loot run system. Loot pools reset every **Friday at 19:00 UTC**. The mod submits regional loot data, and the backend validates, deduplicates, and stores it organized by ISO week.

## Endpoints

Loot pool endpoints are auto-generated by `BasePoolBlueprint('lootpool', Collection.LOOT)`:

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/lootpool/items` | `write:lootpool` + mod | Submit loot pool data |
| GET | `/api/lootpool/items` | `read:lootpool` | Processed current week items |
| GET | `/api/lootpool/current` | `read:lootpool` + mod | Raw current week data |
| GET | `/api/lootpool/all` | `read:lootpool` | Paginated all weeks (max 5 per page) |
| GET | `/api/lootpool/{year}/{week}` | `read:lootpool` | Specific week data |

## Submission Flow

```
POST /api/lootpool/items
    |
    v
base_pool_service.save(Collection.LOOT, data)
    |
    +-- For each region in payload:
    |   +-- Check modVersion >= MIN_SUPPORTED_VERSION
    |   +-- For each item in region:
    |   |   +-- Validate timestamp is within current loot week
    |   |   +-- Strip sensitive fields (playerName, modVersion, timestamp)
    |   +-- Skip region if no valid items remain
    |
    v
enqueue(CollectionRequest(type=LOOT, items=valid_regions))
    |
    v
Queue Worker -> lootpool_repo.save()
```

### Validation Rules

1. **Mod version:** each region must pass the version gate
2. **Timestamp:** each item's timestamp must fall within the current loot week window (Friday 19:00 UTC to next Friday 19:00 UTC)
3. **Sensitive data:** `playerName`, `modVersion`, and `timestamp` are stripped from individual items before storage (the region-level timestamp is preserved)

## Storage and Deduplication

`BasePoolRepo.save()` handles deduplication per `(region, year, week)`:

### Week Calculation

The ISO week and year are computed from the payload's timestamp using `get_lootpool_week_for_timestamp()` with `reset_hour=19`.

### Replacement Rules

When a document already exists for the same `(region, year, week)`:

| Condition | Action |
|-----------|--------|
| New payload has **more items** than existing | Replace |
| Existing doc is **stale** (>1 hour old) AND new has >= items | Replace |
| Otherwise | Skip (keep existing) |

Replacement is done via `delete_one()` + `insert_one()` rather than `replace_one()`.

### Fresh Insert

If no existing document matches, the new document is inserted directly.

## Processed View (GET /api/lootpool/items)

`lootpool_repo.fetch_lootpool()` returns a richly structured view of the current week's data through a complex aggregation pipeline:

### Pipeline Stages

1. **Match** current year/week
2. **Map items** to assign group and type fields:
   - `AspectItem` -> group "Aspect"
   - Items with type containing "TOME" -> group "Tomes"
   - All others -> group is the item's rarity (capitalized)
   - For `PowderItem`/`AmplifierItem`: type is derived from the item name (first two words joined, e.g., "Thunder Powder" -> "ThunderPowder")
3. **Compute effectiveShiny**: `shiny == true AND shinyStat != null`
   - Items marked shiny but without a `shinyStat` value are NOT treated as shiny
4. **Unwind** items array
5. **Group by** `(region, group, effectiveShiny)`, collecting items into lists
6. **Sort items** within each group by `(itemType, name, tier, amount)`
7. **Group by region**, assembling `itemsByGroup` array
8. **Label shiny groups** as "Shiny" in the output
9. **Sort groups** by priority:

| Priority | Group |
|----------|-------|
| 0 | Shiny |
| 1 | Aspect |
| 2 | Mythic |
| 3 | Fabled |
| 4 | Legendary |
| 5 | Rare |
| 6 | Set |
| 7 | Unique |
| 8 | Tomes |
| 9 | Common |
| 10 | Misc |

10. **Sort regions** alphabetically

### Output Structure

```json
[
    {
        "region": "Corkus",
        "week": 11,
        "year": 2026,
        "timestamp": "...",
        "region_items": [
            {
                "group": "Mythic",
                "loot_items": [
                    {
                        "name": "Divzer",
                        "type": "Bow",
                        "rarity": "Mythic",
                        "itemType": "Weapon",
                        "amount": 1,
                        "shiny": false,
                        "shinyStat": null,
                        "icon": "...",
                        "tier": null
                    }
                ]
            }
        ]
    }
]
```

## Raw View (GET /api/lootpool/current)

Returns unprocessed documents from the `lootpool` collection for the current week, with minimal transformation.

## Historical Views

### All Weeks (GET /api/lootpool/all)

Uses `build_pool_pipeline()` which:
1. Unwinds items
2. Copies `type` to `subtype`
3. Groups by `(year, week, region)` collecting items
4. Groups by `(year, week)` collecting regions
5. Sorts by year/week descending (newest first)
6. Applies skip/limit pagination (max 5 weeks per page)

### Specific Week (GET /api/lootpool/{year}/{week})

Same pipeline as above but with a `$match` on the specific year/week. Returns a single document.

## Debug Logging

When a submission arrives within 10 minutes of a pool reset (determined by `is_in_reset_window()`), the raw payload is saved to the `lootpool_debug_logs` collection. These documents auto-expire after 7 days via a TTL index.
